# Agents

Applies the [trunk](../PHILOSOPHY.md) to working with agents. Building *for*
agents is a software concern and stays out of the philosophy for now; this
area is about the agents that work for me. An agent is a tool. None of
[people.md](people.md) applies to it. But it's a tool unlike the others,
powerful enough that integrating it is a craft of its own and young enough
that the craft keeps changing.

1. **An extension of myself.** The agent is never a thing of its own. I adopt
   its output as my output and I own it entirely: the quality bar, the
   verification, the consequences. "The agent did it" doesn't exist. Whatever
   ships under my name is mine, whoever wrote the first draft.
2. **Mastery is the work.** A tool this powerful earns deliberate integration.
   Put real thought into where agents enter the workflow, what they're trusted
   with, and how to wield them better. Adopt the tool's vision applies in
   full: learn how the agent works best instead of forcing old habits onto it.
3. **The grain hasn't settled.** Few strong tools says stop chasing new
   things. Agents are the exception, because their capabilities move too fast
   for "what works now" to stay true. Experimenting is how mastery stays
   current here, and settling on what seems to work today is how it rots.
   This is where creativity shines: keep trying things that might not work.
4. **Add and multiply.** One agent is capacity. Spawning several in parallel
   is addition. Agents that spawn agents is multiplication. Both shapes exist
   to be used: split the work that parallelizes, and hand the orchestration
   itself to an agent when it doesn't.
5. **Materialize, or it rots.** Ship-or-rot at the scale of a single run.
   Whatever lives only in an agent's context dies with the agent, so every
   run ends in an artifact. The artifact takes whatever form fits: code,
   text, an image, an app, an issue filed. Value produced but never
   materialized was never produced at all.
